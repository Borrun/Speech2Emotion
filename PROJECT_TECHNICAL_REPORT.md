# Speech2Emotion 技术方案与实现说明

## 1. 项目定义

`Speech2Emotion` 不是传统的整句情绪分类系统，而是一个面向角色表情驱动的连续情绪建模系统。

系统输入：
- 单通道语音
- 可选文本

系统输出：
- 逐帧情绪类别 `type`
- 逐帧情绪强度等级 `level`
- 逐帧情绪切换边界概率 `boundary`
- 经平滑与约束后的情绪段结果，可直接映射到眼睛模板曲线


## 2. 这是一个什么类型的问题

从机器学习角度看，本项目属于以下几类问题的组合：

### 2.1 时序序列标注

模型不是为整段语音输出单一标签，而是在 `30 fps` 的时间轴上进行逐帧预测，因此本质上是：
- `frame-wise sequence labeling`
- `temporal classification`

### 2.2 多任务学习

模型同时预测三类目标：
- `type`：情绪类型
- `level`：情绪强度等级
- `boundary`：情绪是否在当前帧发生切换

因此训练形式是一个典型的 `multi-task learning` 问题。

### 2.3 在线边界检测

项目最终要服务实时表情驱动，因此模型和后处理都需要支持流式判断“何时切段”。这一部分属于：
- `online boundary detection`
- `streaming temporal decision`

### 2.4 音频-文本融合

音频负责声学时序变化，文本负责语义先验和纠偏，因此整体还带有：
- `audio-text multimodal fusion`

### 2.5 情绪到动画控制映射

最终目标不是只输出分类结果，而是映射为眼睛曲线模板，因此也可以视为：
- `emotion-to-animation control`


## 3. 总体方法概述

整体方案可概括为：

> 使用因果声学特征和因果 TCN 建模逐帧情绪轨迹，联合预测情绪类别、强度和边界，再通过边界解码、平滑约束与文本先验融合，输出稳定的情绪段用于表情控制。

工程链路如下：

1. 连续情绪曲线标注
2. 曲线离散为逐帧监督
3. 声学模型联合预测 `type / level / boundary`
4. 后处理进行边界确认与标签平滑
5. 文本模块提供边界先验和情绪约束
6. 最终结果映射到眼睛模板曲线并可视化


## 4. 数据定义与标注方案

### 4.1 标注单位

标注文件位于 `annotater/labels_new.jsonl`，每条样本包含一条连续曲线 `curve`，每个控制点包含：
- `t`：时间戳
- `type`：情绪类型
- `value`：情绪强度值

项目采用的是“连续曲线标注”而不是“整段单标签标注”，这样可以表达一段语音内部的情绪转移，更适合动画驱动。

### 4.2 情绪类别

当前使用 `10` 类情绪：

- `happy`
- `sad`
- `angry`
- `fear`
- `calm`
- `happy_confused`
- `sad_confused`
- `angry_confused`
- `fear_confused`
- `calm_confused`

这是一套“基础情绪 + 疑问/不确定语气”的扩展标签体系，用来提升对真实对话语气的覆盖能力。

### 4.3 强度建模

强度没有直接做连续回归，而是离散为 `6` 档 `0~5`，便于稳定训练和直接驱动模板：

| level | value 区间 |
| --- | --- |
| 0 | `0 ~ 10` |
| 1 | `11 ~ 25` |
| 2 | `26 ~ 50` |
| 3 | `51 ~ 85` |
| 4 | `86 ~ 110` |
| 5 | `111 ~ 150` |


## 5. 预处理与特征工程

### 5.1 音频标准化

训练和推理统一使用：
- 采样率：`16 kHz`
- 声道：单声道

若输入音频不是 `16 kHz`，会在加载阶段做重采样。

### 5.2 声学特征

特征提取器采用 `CausalLogMelFeaturizer`，关键参数如下：

| 参数 | 取值 |
| --- | --- |
| `sample_rate` | `16000` |
| `n_mels` | `80` |
| `hop_sec` | `1 / 30` |
| `win_sec` | `0.05` |
| `f_min` | `40` |
| `f_max` | `7600` |
| `center` | `False` |

设计动机：
- `hop_sec = 1/30`：直接和动画帧率对齐，输出天然是 `30 fps`
- `win_sec = 50 ms`：兼顾短时稳定性与低延迟
- `center = False`：严格因果，不引入未来帧信息，适合在线部署

### 5.3 标签离散化

数据集在训练前会将连续曲线离散成逐帧标签：
- `y_type[t]`
- `y_lvl[t]`
- `y_bnd[t]`

其中：
- `y_type` 由 `type_at(curve, t)` 决定
- `y_lvl` 由 `value_at_step(curve, t)` 决定，再映射到 `0~5`
- `y_bnd[t]=1` 表示相邻帧在 `type` 或 `level` 上发生变化

这一步将“动画曲线问题”转成了“逐帧监督学习问题”。


## 6. 模型结构

### 6.1 主干网络

主模型为 `EmotionTCN`，整体结构为：

`Audio -> Causal Log-Mel -> Causal TCN -> {Type Head, Level Head, Boundary Head}`

### 6.2 网络配置

当前训练完成的 `10` 类模型使用如下结构：

| 模块 | 配置 |
| --- | --- |
| 输入维度 | `80` 维 log-mel |
| 通道数 | `128` |
| TCN 层数 | `6` |
| 卷积核大小 | `3` |
| dilation | `1, 2, 4, 8, 16, 32` |
| dropout | `0.1` |
| 情绪类别数 | `10` |
| 强度等级数 | `6` |
| 边界头 | `开启` |

### 6.3 TCN Block 形式

每个时序块是一个残差因果块：

1. `CausalConv1d`
2. `ReLU`
3. `Dropout`
4. `1x1 Conv`
5. `ReLU`
6. 残差相加

这种结构兼顾了：
- 因果性
- 时域膨胀感受野
- 轻量级部署复杂度

### 6.4 输出头

模型末端包含三个并行 head：
- `type_head`：输出 `[B, T, 10]`
- `lvl_head`：输出 `[B, T, 6]`
- `bnd_head`：输出 `[B, T]`

### 6.5 感受野说明

按当前 `6` 层、`kernel=3`、`dilation=1/2/4/8/16/32` 的结构推算，主干网络的有效历史感受野约为：

- `127` 帧
- 约 `4.23` 秒历史上下文（以 `30 fps` 计）

这意味着模型在不看未来信息的前提下，已经能利用约 `4` 秒的历史语境来判断当前情绪状态。

### 6.6 参数规模

当前 `10` 类配置模型参数量约为：

- `407,313` 参数

这使其在保持较低部署成本的同时，具备足够的时序建模能力。


## 7. 训练框架与优化策略

### 7.1 训练框架

训练框架基于：
- `PyTorch`
- `torch.utils.data.DataLoader`
- `torchaudio`

优化器使用：
- `AdamW`

### 7.2 默认训练超参数

训练脚本 `train/train_emotion_tcn.py` 的默认超参数如下：

| 超参数 | 取值 |
| --- | --- |
| `epochs` | `80` |
| `batch_size` | `4` |
| `lr` | `2e-4` |
| `weight_decay` | `1e-3` |
| `seed` | `1234` |
| `val_ratio` | `0.2` |
| `channels` | `128` |
| `layers` | `6` |
| `dropout` | `0.1` |
| `label_smoothing` | `0.05` |
| `w_type` | `1.0` |
| `w_lvl` | `0.7` |
| `w_bnd` | `0.3` |
| `grad_clip` | `1.0` |
| `pos_weight_cap` | `20.0` |

### 7.3 损失函数

训练目标为多任务加权损失：

`L = w_type * CE(type) + w_lvl * CE(level) + w_bnd * BCE(boundary)`

具体设置：
- `type`：交叉熵，带 `label_smoothing=0.05`
- `level`：交叉熵，带 `label_smoothing=0.05`
- `boundary`：`BCEWithLogitsLoss`

### 7.4 边界不平衡处理

边界帧远少于非边界帧，因此训练时引入 `pos_weight` 处理类别不平衡。

当前 `10` 类 checkpoint 中统计到：
- 边界正样本比例 `pos_ratio = 0.007706`
- 理论 `raw_pos_weight = 128.77`
- 实际截断后 `pos_weight_used = 20.0`

这样做的目的，是在召回边界的同时避免边界头发生明显的过预测。

### 7.5 训练稳定性策略

为保证训练更稳定，训练中还使用了：
- 固定随机种子
- 验证集划分 `val_ratio=0.2`
- `grad_clip=1.0`
- 多任务损失加权

## 9. 推理与后处理

### 9.1 原始逐帧输出

模型逐帧输出：
- `type_id`
- `level_id`
- `boundary_p`

原始输出能反映模型瞬时判断，但通常会存在局部抖动，不适合直接驱动动画。

### 9.2 边界解码

推理端会先将 `boundary_p` 转为情绪切换点。当前同步到 Python 侧的默认参数为：

| 参数 | 取值 |
| --- | --- |
| `thr_on` | `0.12` |
| `thr_off` | `0.06` |
| `confirm_win` | `3` |
| `min_gap` | `5` |

含义是：
- 先通过上阈值触发候选边界
- 通过下降阈值或确认窗口完成事件确认
- 通过 `min_gap` 避免过密切段

### 9.3 标签平滑

在边界确认后，还会对 `(type_id, level_id)` 做与 `cpp_emotion` 一致的平滑：

| 参数 | 取值 |
| --- | --- |
| `smooth_win` | `3` |
| `hysteresis` | `2` |

具体策略包括：
- 小窗口多数投票平滑
- hysteresis 防抖
- 分段后 segment-majority 回填

这一步的本质是把“逐帧分类结果”变成“稳定的情绪段”，使结果更适合作为动画控制信号。

### 9.4 文本约束

在线链路中，文本模块还会提供两类信息：
- 文本边界先验
- 文本情绪分布约束

当前 `10` 类文本约束已经支持将疑问、迟疑、不确定等语义偏向映射到：
- `happy_confused`
- `sad_confused`
- `angry_confused`
- `fear_confused`
- `calm_confused`

因此最终输出不只是“声学分类结果”，而是“声学结果 + 语义先验 + 平滑约束”的联合结果。

## 13. 方案特点与优势

相较于普通的整句情绪分类方案，本项目的特点在于：

1. 目标不是单标签分类，而是连续情绪轨迹建模。
2. 模型不是单一分类头，而是 `type + level + boundary` 多任务联合优化。
3. 主干网络是严格因果结构，天然适配在线部署。
4. 文本不是独立旁路，而是作为边界和情绪的先验约束进入系统。
5. 最终输出能直接映射到眼睛曲线，形成完整动画驱动链路。


## 14. 下一步优化方向

后续可以重点从以下几个方面继续提升：

1. 扩充 `*_confused` 复合类样本，缓解长尾类别不平衡。
2. 优化边界头训练策略，例如更细的 hard negative 采样或 focal 类损失。
3. 在文本融合阶段加入更稳定的上下文情绪记忆。
4. 继续做 ONNX 到 C++ 推理侧的一致性验证。
5. 将当前 Python 侧验证通过的平滑逻辑固化为统一线上版本。


## 15. 结论

`Speech2Emotion` 的核心价值，不在于做一个“语音情绪分类器”，而在于构建一条完整的“语音到连续情绪状态，再到表情控制参数”的工程链路。

目前项目已经完成：
- `10` 类连续情绪标注体系
- 因果时序模型训练
- 多任务预测
- 文本约束融合
- 平滑与边界解码
- 眼睛曲线驱动可视化
- ONNX 导出与部署准备

从技术路线看，这是一套兼顾研究可解释性、工程可落地性和实时部署需求的方案。
