一、系统在做什么（本质逻辑）

你这个系统干了三件事：

1️⃣ 音频流 → 连续情绪轨迹

用 TCN 从语音里预测：
	•	每一帧：
	•	情绪类型（happy / sad / angry…）
	•	情绪强度
	•	是否是“边界”（情绪切换点）

👉 这一步是纯声学建模
👉 对应：EmotionTCN  ￼

⸻

2️⃣ 文本 → 先验 + 纠偏

文本不直接分类，而是：

两个作用：

（1）预测“哪里可能断句”
→ 标点 / 情绪翻转
→ 变成概率分布 p_text(t)

（2）给“情绪标签”加约束
→ 比如：
	•	文本明显正向 → 不可能是 sad
	•	有问号 → 更可能 confused

👉 对应：TextPrior + TextEmotionConstraint  ￼

⸻

3️⃣ 音频 + 文本融合 → 情绪段

核心不是分类，而是：

找“情绪变化的时间点”

通过：

p_fused(t) = 0.8 * 音频 + 0.2 * 文本

再用一个状态机来决定：
	•	什么时候真的发生了情绪切换
	•	防止抖动 / 误触发

👉 对应：OnlineBoundaryDetector  ￼

端到端多模态 Transformer 版本


可以。把你现在这套“音频主导 + 文本先验/纠偏”的系统，升级成端到端多模态 Transformer，核心变化是：

不再把“音频边界检测、文本先验、规则融合”拆开手工做，

	•	什么时候情绪切换

1. 先说目标：端到端版到底要替代什么

你现在的系统是：
	•	音频特征提取
	•	TCN 做帧级 type / level / boundary
	•	文本做边界先验
	•	文本再做规则纠偏
	•	状态机切段
	•	段内投票

端到端 Transformer 版的目标是把这些模块尽量收敛成：

Audio Encoder
   + 
Text Encoder
   +
Cross-modal Fusion / Joint Encoder
   ↓
Frame-wise Emotion Head
Boundary Head
Segment/Utterance Head

也就是让模型自己学会：
	•	声学情绪模式
	•	文本语义情绪模式
	•	两者对齐关系
	•	边界触发逻辑

⸻

2. 推荐的总体架构

最适合你这个任务的，不是“单纯把文本和音频拼起来”，而是三段式：

架构总览

原始音频 wave
   ↓
Audio Frontend
   ↓
Audio Encoder (causal / streaming transformer)

文本 sentence / token
   ↓
Text Encoder (BERT / RoBERTa / MacBERT / multilingual encoder)

音频序列表示 + 文本序列表示
   ↓
Cross-Modal Fusion Transformer
   ↓
多任务输出头：
  1) frame emotion head
  2) frame intensity head
  3) boundary head
  4) optional segment head

3. 输入怎么设计

⸻

3.1 音频输入

你现在是 LogMel，这个可以保留，没必要一开始就上 raw waveform。

推荐：
	•	16kHz mono
	•	80 维 log-mel
	•	25ms 或 40ms window
	•	10ms 或 20ms hop

如果你追求和现有系统兼容，也可以继续保持 50ms window + 33ms hop。

两种做法

方案 A：log-mel + Transformer
优点：
	•	稳
	•	好训
	•	算力可控
	•	容易部署

方案 B：预训练语音模型
比如：
	•	wav2vec 2.0
	•	HuBERT
	•	WavLM

优点：
	•	表达能力更强
	•	小数据下通常更好

如果你是工程落地，我建议：

先用 log-mel + streaming transformer 起一个版本，
再尝试把 audio encoder 换成 HuBERT/WavLM 的轻量版。

⸻

3.2 文本输入

文本最好不是整句扔进去就完，而是做 token 级输入。

例如：

“我本来很高兴，但是后来真的有点害怕。”

编码成：
	•	token ids
	•	attention mask
	•	optional punctuation mask
	•	optional sentiment lexicon mask

文本 encoder 可以选：
	•	中文：MacBERT / RoBERTa-wwm
	•	英中混合：XLM-R
	•	英文：RoBERTa / DeBERTa

⸻

3.3 对齐信息

端到端模型里最关键的问题之一就是：

文本和音频怎么对齐？

有三种层级：

方案 1：弱对齐

只有整句文本，没有 token timestamp。

做法：
	•	音频序列长度 = T
	•	文本序列长度 = L
	•	用 cross-attention 让音频帧自动关注文本 token

这个最简单，也是最现实的起点。

⸻

方案 2：半监督对齐

有 ASR token 时间戳。

做法：
	•	每个 token 有 start/end time
	•	构建 token-to-frame alignment mask
	•	cross-attention 时加位置偏置

这会明显提升“边界”和“情绪翻转”的精度。

⸻

方案 3：联合 ASR + Emotion

模型同时做：
	•	ASR
	•	Emotion
	•	Boundary

这个最强，但训练和部署复杂很多。你当前阶段不建议第一版就上。

⸻

4. 核心模型结构怎么搭

⸻

4.1 Audio Encoder

这里建议不要直接用标准 Transformer Encoder 全局看未来，除非你只做离线。

如果你还要实时，就要做 streaming / causal transformer。

推荐结构

LogMel [B, T, 80]
  ↓
Linear / Conv subsampling
  ↓
Positional Encoding
  ↓
N layers Causal Transformer Encoder
  ↓
A_audio [B, T, D]

关键点
	•	causal self-attention
	•	chunk-based attention
	•	limited left context
	•	optional memory cache

这样才能流式推理。

⸻

4.2 Text Encoder

Tokens [B, L]
  ↓
Pretrained LM
  ↓
A_text [B, L, D]

这里文本 encoder 通常可以：
	•	直接微调
	•	或前几层冻结，减少过拟合

⸻

4.3 融合层：最关键

这部分决定多模态是不是“真的融合”。

我推荐三种方式里选第二种。

⸻

方式 A：简单拼接

把文本句向量池化成一个全局向量，再拼到每一帧音频后面。

[audio_t ; text_cls]

优点：
	•	简单
	•	快

缺点：
	•	太粗
	•	边界效果一般

⸻

方式 B：Cross-Attention 融合（推荐）

Q = audio frames
K,V = text tokens

意思是：

每一帧音频，都去看“和哪些文本 token 最相关”

这样模型可以学到：
	•	某一段声调对应哪个文本词
	•	问号、否定词、转折词对情绪的影响
	•	“但是 / 不过 / 突然 / really / why” 这类翻转点

这和你现在的 TextPrior 思路最接近，但更强。

⸻

方式 C：双向融合

不仅 audio attend text，text 也 attend audio，再做 joint encoder。

Audio Encoder
Text Encoder
↓
Bi-directional Cross-modal Blocks
↓
Joint sequence

最好，但更重。

如果你追求性能上限，可以用。
如果追求在线落地，先别太复杂。

⸻

5. 输出头怎么设计

端到端版最好做多任务学习，而不是只做一个类别分类。

⸻

5.1 帧级情绪分类头

H_t → Linear → softmax over emotion classes

输出：
	•	happy
	•	sad
	•	angry
	•	fear
	•	calm
	•	confused
	•	neutral / other（建议加）

我建议加一个 neutral，不然很多过渡帧会被硬塞进某个情绪类。

⸻

5.2 帧级强度头

H_t → Linear → 6-level intensity

或改成回归：

H_t → scalar in [0,1]

如果你业务需要离散等级，继续分类即可。

⸻

5.3 边界检测头

H_t → Linear → sigmoid

输出每帧是否是边界概率。

这相当于把你现在的 bnd_head + detector 保留，但 detector 不再依赖手工文本先验。

⸻

5.4 段级情绪头（可选）

如果有 segment 标注，可以再加一个段级头：
	•	对 segment 内帧做 pooling
	•	输出段级 emotion

这样能约束模型学得更稳定。

⸻

6. 训练标签怎么定义

这是最关键的工程问题之一。

⸻

6.1 帧级标签从哪来

如果你手里只有“句子级情绪标签”，端到端边界很难学好。
最好要有：
	•	段级边界标签
	•	段级 emotion 标签

再投影成帧级标签：

情绪帧标签

在 segment 区间内，所有帧继承该 segment emotion。

边界帧标签

在边界附近 ±k 帧标 1，其余标 0。

例如：
	•	真正边界在第 100 帧
	•	则 [98, 99, 100, 101, 102] 可视为 positive

这样比单点监督更稳定。

⸻

6.2 如果没有精确边界标签

可以这样做：

方法 A：弱监督

用现有系统的 BoundaryEvent 作为 pseudo label。
这是非常实用的迁移路径。

方法 B：CTC/segmentation 风格监督

只知道段顺序，不知道精确边界，利用动态规划去学 latent segmentation。

这个更学术，但实现复杂。

⸻

7. Loss 怎么设计

推荐总损失：

L = λ1 * L_emotion
  + λ2 * L_intensity
  + λ3 * L_boundary
  + λ4 * L_segment
  + λ5 * L_alignment(optional)


⸻

7.1 Emotion loss

帧级交叉熵：

CE(frame_emotion, y_emotion)

如果类别不平衡，建议：
	•	class-balanced CE
	•	或 focal loss

⸻

7.2 Intensity loss
	•	分类：CE
	•	回归：MSE / SmoothL1

⸻

7.3 Boundary loss

边界非常稀疏，强烈建议不要直接普通 BCE。

推荐：
	•	focal BCE
	•	dice loss
	•	BCE + temporal smoothing

比如：

L_boundary = FocalBCE(boundary_pred, boundary_label)


⸻

7.4 Temporal consistency loss

为了减少抖动，可以加时序平滑约束：

L_smooth = Σ ||p_t - p_{t-1}||

或者对非边界区加 stronger consistency：

在没有边界的地方，相邻帧情绪分布不要乱跳。

这相当于把你现在的“多数投票 + 迟滞”变成训练目标的一部分。

⸻

7.5 Cross-modal consistency loss（可选）

如果文本明确表达正向、音频却预测成强烈 sad，可以给一个一致性约束。

不是硬规则，而是 soft regularization。

⸻

8. 在线版本怎么做

如果你要保留实时能力，端到端 Transformer 一定不能做成纯离线大模型。

⸻

8.1 推荐：chunk-based streaming transformer

输入方式

每次输入一个 chunk，例如：
	•	0.5s 音频
	•	带上左侧缓存 context

模型结构：
	•	self-attention 只看当前 chunk + 历史缓存
	•	不看未来

这样就等价于 Transformer 版的“因果 TCN”。

⸻

8.2 文本怎么接入在线流

有两种现实方案：

方案 A：整句文本一次性给定

这和你当前文档一致。
适合：
	•	用户先给文本
	•	然后音频边说边解码

方案 B：ASR 增量文本

边 ASR 边更新 token 序列。

这时候 text encoder 也要支持增量更新。
复杂度高，但更真实。

⸻

8.3 边界输出怎么稳定

即使端到端，在线部署时仍建议保留一个轻量解码器，而不是直接 p > 0.5。

也就是：
	•	模型输出 boundary probability
	•	外部仍保留一个小状态机
	•	on threshold
	•	off threshold
	•	min gap
	•	confirm frames

因为这不是“非端到端”，而是推理后处理，非常合理。

9. 一个推荐的具体模型蓝图

我给你一个更落地的版本。

⸻

9.1 Backbone

Audio branch
	•	80-d logmel
	•	2 层 conv subsampling
	•	6 层 causal conformer / transformer
	•	hidden dim 256

Text branch
	•	6 层 BERT-base 截断版，或轻量中文 encoder
	•	hidden dim 256

Fusion
	•	4 层 cross-attention block
	•	audio attends text
	•	optional FFN + residual + norm

Heads
	•	emotion head: 7/8 类
	•	intensity head: 6 类
	•	boundary head: 1 sigmoid

⸻

9.2 输出

Audio frames: T
Text tokens: L

A_audio: [B,T,256]
A_text : [B,L,256]

Fused : [B,T,256]

Outputs:
emotion_logits  [B,T,C]
intensity_logits[B,T,K]
boundary_logits [B,T]

11. 端到端版的优势

⸻

优势 1：融合更自然

不需要手写：
	•	文本权重 0.2
	•	sigma=4
	•	sentiment_thr=0.3

模型自己学。

⸻

优势 2：对复杂语义更强

例如：
	•	转折
	•	反问
	•	语气和语义不一致
	•	文本正向但声音压抑

这类情况手工规则往往不够。

⸻

优势 3：可统一优化

当前系统是模块化的，局部最优不一定全局最优。
端到端可以直接对最终目标优化。

12. 端到端版的代价

⸻

代价 1：更吃数据

这是最大的现实问题。

你现在的系统很多能力来自人工结构设计。
端到端模型要学会这些，需要更多标注数据。

⸻

代价 2：可解释性下降

以前你能说：
	•	这里切段因为标点 + 音频峰值
	•	这里改成 confused 因为问号

所以最佳方案不再是纯 causal TCN / causal Transformer，

三、为什么这里应该用“非因果”而不是“因果”

因为你的业务目标不是“尽快预测下一帧”，而是：

	•	后文

	•	转折后的真实情绪走向

这对边界和情绪类型都很重要。

前半句如果只看局部可能偏 happy，

	•	“挺开心的”是不是只是过渡

因为你的输出轴是 frame 轴，不是 token 轴。

五、你这个场景下，最关键的不是“推理”，而是“发射策略”

1. 计算时间

模型可以提前离线完成

2. 发射时间

所以系统不是：

先算出全帧结果，再按播放时钟逐帧发射

   ↓

  1. 提取全帧 Mel

  3. 生成 frame-wise emotion / intensity / boundary
  4. 可选做一次平滑和边界解码
  5. 写入 timeline buffer

播放线程：
  按主轴 frame_idx 读取 timeline buffer[t]
  同步喂给：
    - 帧级节奏播放器
    - 上层实时音频播放器的情绪状态

这个是最稳的。

所以最合适的系统，不是纯因果，也不是纯离线，而是：

增量可修正的准在线系统（incremental non-causal / lookahead-augmented online inference）

一、现在把你的真实约束重新准确定义

时间线上有 4 条并行链路

从 t=0 同时开始：

1. TTS 音频传入链路
	•	持续把音频数据送进系统
	•	但速度比真实播放快
	•	所以系统看到的音频进度 audio_ingest_time 会逐渐领先于 playback_time

2. 主音频播放器
	•	按真实时间播放
	•	是用户真正听到的时间轴

3. 帧级节奏播放器
	•	和主轴对齐
	•	需要精确 frame-level 对齐输出

4. 情绪模型
	•	输入是“当前已传入的音频前缀 + 全文本”
	•	输出是对部分时间轴的情绪估计
	•	随着未来音频不断到达，可以对尚未播出甚至刚播出的区域做修正

⸻

二、这个约束的核心含义

现在最关键的一句话是：

模型在播放时，并不是只能看过去；它能看多少未来，取决于此刻音频传入已经领先播放多少。

所以你这里的“实时性”其实不是固定延迟实时，而是：

带动态前视窗口的实时对齐系统

定义几个量会很清楚：

⸻

1. 播放位置

设当前播放到主轴帧：

t_{play}

⸻

2. 已传入音频位置

设当前系统已经收到的最远音频帧：

t_{ingest}

通常有：

t_{ingest} \ge t_{play}

并且这个差值是动态变化的。

⸻

3. 可用前视窗口

定义：

L(t) = t_{ingest} - t_{play}

这就是当前时刻模型能看到的“未来帧数”。
	•	如果 L(t)=0，退化成严格在线
	•	如果 L(t)=30，说明能看到未来 30 帧
	•	如果 L(t) 很大，局部上接近离线

这就是你系统设计的核心变量。

⸻

三、因此模型设计目标要改成什么

现在不能再设计成：

方案 A：纯因果

因为你明明在播放过程中能逐步拿到未来音频，纯因果会浪费信息。

也不能设计成：

方案 B：全离线

因为完整音频到齐的时刻不确定，而且播放早就开始了。

所以应该设计成：

prefix-to-timeline 的增量推理系统

也就是：
	•	输入：截至当前时刻收到的音频前缀
	•	输出：该前缀范围内的帧级情绪轨迹
	•	对靠近当前 ingest 末端的区域，可以先给 provisional 结果
	•	随着更多未来音频到达，再对“尚未稳定”的时间段回写修正

⸻

四、这会直接决定输出语义必须分层

你不能只输出一个 emotion[t]，而要区分两类结果：

1. 已提交结果（committed）

这些结果已经足够稳定，不再修改，或者极少修改。

用于：
	•	帧级节奏播放器
	•	对外展示
	•	下游控制逻辑

⸻

2. 暂定结果（provisional / speculative）

这些结果是模型根据当前可见前缀做的预测，之后允许修正。

用于：
	•	内部缓存
	•	预览
	•	提前准备后续段的动作

⸻

关键思想

对某个帧 τ，如果它距离当前 ingest 末端太近，那么：
	•	未来上下文还不够
	•	边界可能漂
	•	情绪类别可能翻转

所以这类帧不能立刻 hard commit。

⸻

五、最合理的落地机制：滑动稳定区 + 可修正区

这个是你场景的核心设计。

设当前已收到音频到 t_ingest。
模型每次推理后，把 [0, t_ingest] 分成两段：

1. 稳定区（stable zone）

[0, t_{stable}]

这一段离 ingest 尾部足够远，未来上下文已经够多，可以提交。

例如定义：

t_{stable} = t_{ingest} - R

其中 R 是你要求的最小右侧上下文帧数。

⸻

2. 可修正区（revision zone）

(t_{stable}, t_{ingest}]

这一段虽然有预测，但不能承诺最终不变。
等后续更多音频到达，再重新推理并修正。

⸻

直觉上

你是在维护一个不断向前推进的“稳定边界”。

只要某一帧已经获得足够的未来上下文，它的结果就从 provisional 变成 committed。

⸻

六、于是，最适合你的模型不是纯 streaming transformer，而是“有限右上下文 Transformer”

这是架构结论。

推荐模型形态

Chunked Transformer / Conformer with bounded right context

也就是：
	•	对每个目标帧，不一定只看过去
	•	允许看有限未来，比如 R=30 帧
	•	未来上下文不要求完整全局，只要求一段右窗
	•	一旦右窗满足，就可以稳定输出

⸻

为什么这比双向全局更好

如果做全局双向 Transformer，那么每多来一点新音频，理论上整个后半段都可能变化，缓存和增量更新代价很高。

如果做有限右上下文，则有个很好用的性质：

某帧一旦拿到足够右上下文，它的表示就稳定了。

这正适合你的“提交/修正”机制。

⸻

七、因此最推荐的模型结构变成这样

1. 文本分支

文本一次输入，全文可见。

这个不变：
	•	Text Encoder 一次编码
	•	输出 token memory [L, D]
	•	后续整个播放期间复用

⸻

2. 音频分支

音频是增量到达的，但音频 encoder 不用纯因果，而用：

左全局 / 左长上下文 + 右有限上下文

例如每帧可看：
	•	左侧：全部已到达历史
	•	右侧：最多 R 帧未来

这可通过以下方式实现：
	•	chunk-based attention
	•	blockwise conformer
	•	local window transformer
	•	latency-controlled bidirectional transformer

⸻

3. 多模态融合

仍然推荐：
	•	audio frames 作为 query
	•	text tokens 作为 key/value

因为文本全局已知，正适合作为稳定 memory。

⸻

4. 输出头

保持三头：
	•	frame emotion
	•	frame intensity
	•	boundary

⸻

八、推理机制应该怎么落地

现在重点不只是模型，而是增量推理逻辑。

⸻

推理循环

每当新音频 chunk 到达时：

Step 1

更新音频特征缓存到 t_ingest

Step 2

对最近一段需要更新的时间窗做增量推理

不是每次全量重算全历史，而是只重算：

[t_{recompute\_start}, t_{ingest}]

其中 t_recompute_start 至少要覆盖：
	•	右上下文依赖范围
	•	边界修正传播范围
	•	平滑器影响范围

⸻

Step 3

输出两类结果：
	•	committed frames: 新增稳定帧
	•	provisional frames: 靠近尾部的暂定帧

⸻

Step 4

播放线程只消费：
	•	已经到播放时刻的 committed 结果
	•	如果 committed 还没准备好，再退而求其次用 provisional

不过系统设计上最好保证：

在帧真正播到之前，它已经被提交

⸻

九、这会导出一个非常重要的工程指标：最小提交提前量

因为播放线程需要准时拿到帧结果，所以你必须满足：

t_{stable}(wallclock) > t_{play}(wallclock)

也就是：

稳定区推进速度，必须始终跑在播放前面。

否则播放器走到某帧时，这一帧还没被 commit，就会出问题。

⸻

这取决于三个东西

1. 音频传入领先量

TTS 传入到底快多少

2. 模型所需右上下文

比如你要 R=20 帧、R=40 帧

3. 推理耗时

每次重算窗口要多久

⸻

工程上可以写成安全条件

若当前可用前视窗口为 L(t)，而模型提交一帧需要右上下文 R，则要有：

L(t) > R + M

其中 M 是推理和调度安全余量。

这条式子会直接决定你的模型可用多大的 lookahead。

⸻

十、对边界检测的影响尤其大

你当前系统里边界是核心。文档中边界检测依赖 bnd_head 和状态机，且还有 confirm_win、min_gap 这类时序约束。 ￼

在新约束下，边界检测最合理的做法不是单纯帧分类，而是：

边界候选 + 延迟确认 + 可回写修正

也就是：

1. 边界候选

模型先输出某段附近的边界概率峰

2. 延迟确认

当右侧未来上下文够了，再确认峰值位置

3. 可回写修正

如果后续发现峰值更合适，可以在尚未 commit 的区域内移动边界

这个机制和你现在的状态机思想兼容，但要从“严格因果确认”改成“有限未来确认”。

⸻

十一、因此你最应该避免的一个误区

不要把它设计成：

每来一个 chunk，就对当下播放帧立刻做最终判决

这是不对的。
因为你明明有越来越多的未来音频，而情绪和边界恰恰很依赖未来收束信息。

正确做法是：

对“播放附近帧”做预测，但只对那些已经获得充分右上下文的帧做最终提交。

⸻

十二、基于这个约束，最合理的总方案

我把它压缩成一个明确版本：

模型层

Latency-controlled multimodal Transformer
	•	Audio encoder：有限右上下文 Transformer / Conformer
	•	Text encoder：全文一次编码
	•	Fusion：audio-to-text cross-attention
	•	Heads：emotion / intensity / boundary

⸻

系统层

Incremental inference with commit/revise buffer

维护一个时间轴缓冲区，每帧有状态：
	•	UNSEEN
	•	PROVISIONAL
	•	COMMITTED

⸻

播放层

三个播放器共享同一主轴 frame_idx，但读取策略不同：
	•	帧级节奏播放器：严格按帧取 committed
	•	上层音频播放器：按播放时钟同步读取当前情绪状态
	•	如需 UI 预览，可读 provisional

⸻

十三、这时“端到端”的正确定义

在你的场景里，“端到端”不应该理解成：

一次性喂完整音频，直接输出最终整条轨迹

而应该理解成：

模型内部统一学习音频、文本、边界、情绪；系统外部允许增量输入、阶段性修正和按播放轴提交。

也就是：
	•	模型端到端
	•	系统不是单次一次性，而是增量可修正

这才符合真实业务。

⸻

十四、最终结论

你这个补充之后，最准确的落地结论是：

模型既不能假设播放前拿到完整音频，也不应该退回纯因果；最佳方案是使用“有限右上下文的多模态 Transformer”，在播放过程中随着音频前视窗口增长，持续对未来一小段和未提交区域进行修正，并仅将已获得足够未来上下文的帧级结果提交给播放器。

⸻

一、系统时序 / 缓存协议

先给结论：

你的系统最适合采用
“增量接收音频 + 周期性局部重算 + 帧级稳定提交 + 播放线程只读提交结果”
的架构。

核心思想是把时间轴上的每一帧分成三种状态：
	•	UNREADY：连音频都还没到，不能算
	•	REVISABLE：已经有预测，但未来还可能修正
	•	STABLE：右侧上下文足够，结果不再改，播放器可直接消费

⸻

1.1 建议的统一主轴

先把所有模块统一到同一个 frame index 主轴。

你文档里当前帧移约为：
	•	hop_length = 533 @ 16kHz
	•	即约 33.3ms / frame
	•	约等于 30 fps  ￼

这个我建议继续保留，理由有三个：

第一，兼容你现有工程

你已有帧级节奏播放器，主轴已经很可能围绕这个粒度在工作。

第二，三播放器同步简单

一旦统一成 frame_idx，所有播放器都只认这条主轴。

第三，边界和情绪都足够用了

情绪段边界本来就是百毫秒级问题，30fps 基本够。

所以系统统一定义：

sample_rate = 16000
frame_hop_sec = 1 / 30
fps = 30

主轴映射：

frame_idx = floor(time_sec * 30)
time_sec   = frame_idx / 30


⸻

1.2 推荐 chunk 大小

这里说的是音频输入 chunk，不是模型帧窗口。

你的约束是：TTS 输出流和播放器同时开始，但音频传入更快。
所以 chunk 大小要兼顾：
	•	输入线程调度成本
	•	特征提取效率
	•	模型重算频率
	•	修正灵敏度

我建议分两层：

⸻

层 1：音频接收 chunk

建议：

audio_ingest_chunk = 160ms ~ 320ms

更具体推荐：

256ms

换算：
	•	0.256s * 16000 = 4096 samples

这个大小非常顺手，工程上也常见。

为什么不是更小，比如 20ms
太碎，调度频率太高，模型会忙于重算。

为什么不是更大，比如 1s
会降低修正和提交频率，影响“边播边稳定输出”。

⸻

层 2：模型触发周期

模型不一定每来一个 chunk 都立刻重算一次。
建议：

infer_tick = 240ms ~ 400ms

推荐：

infer_tick = 320ms

也就是：
	•	接收线程持续写入 PCM buffer
	•	推理线程每 320ms 看一次“现在多了多少新帧”
	•	足够多就触发局部重算

这样吞吐更稳。

⸻

1.3 每次重算窗口多大

这是关键参数。

你不是每次全量重算历史，而是只重算最近一段窗口。
这段窗口必须覆盖：
	•	新到达音频带来的未来上下文影响
	•	边界位置可能回移/前移的范围
	•	平滑器或 refinement 的影响范围

⸻

推荐重算窗口组成

定义：
	•	T_ingest：当前已收到的最远帧
	•	W_left_recompute：向左回看的重算范围
	•	W_right_available：当前新增的右侧范围（即到 T_ingest）

每次重算区间：

[T_ingest - W_left_recompute, T_ingest]


⸻

建议取值

W_left_recompute = 90 ~ 180 frames

按 30fps 换算：
	•	90 帧 ≈ 3 秒
	•	180 帧 ≈ 6 秒

我更建议先用：

W_left_recompute = 120 frames  (≈ 4 秒)


⸻

为什么不是只重算最后 20 帧

因为边界和段级情绪会“向左传播修正”。

例如：
	•	新未来音频到来后
	•	原本以为某段是 happy
	•	后来发现其实只是过渡
	•	真正边界要往前挪 0.5~1.5 秒

如果只重算很短尾部，这类修正做不到。

⸻

为什么不是每次重算整段历史

太贵，而且没有必要。
只要模型是有限右上下文，离尾部足够远的帧已经稳定，不需要再动。

⸻

1.4 多少帧算 stable

这是系统最关键的协议参数之一。

定义一个帧 t 变成 stable 的条件：

它已经获得足够多的右侧未来上下文，且后处理也已完成。

形式化可写成：

stable if T_ingest - t >= R_stable

其中 R_stable 是“提交所需右上下文帧数”。

⸻

推荐 stable 阈值的分层定义

不要只定义一个裸参数，最好拆成三部分：

R_stable = R_model + R_boundary + R_margin

其中：

1. R_model
模型编码这个帧所需的最小右上下文

2. R_boundary
边界确认额外需要的未来帧
因为边界不是分类点，要等峰值和回落更清楚

3. R_margin
工程安全裕量
防止线程调度、缓存滞后、推理抖动

⸻

推荐初始值

如果你用 30fps，我建议第一版先这样：

R_model    = 24 frames   ≈ 0.8s
R_boundary = 12 frames   ≈ 0.4s
R_margin   =  6 frames   ≈ 0.2s
--------------------------------
R_stable   = 42 frames   ≈ 1.4s

也就是：

当某帧右边已经到达了约 1.4 秒的未来音频时，这一帧就可以 commit。

这个量级和你文档里现在 C++ 端 kEmitDelay ≈ 1.5s 很接近，说明这在你的系统里是现实可接受的。 ￼

⸻

可选更激进版本

如果后续发现 TTS 传入通常领先很多，可以降到：

R_stable = 30 frames ≈ 1.0s

如果想让边界更稳，可以升到：

R_stable = 48 ~ 60 frames ≈ 1.6s ~ 2.0s


⸻

1.5 commit / revise buffer 怎么设计

这是系统实现核心。

建议不要只存“当前最终 label”，而是为每一帧建立一个结构体。

⸻

推荐 FrameState 结构

FrameState {
    int frame_idx;

    // 音频可用性
    bool audio_arrived;

    // 生命周期
    enum status {
        UNREADY,     // 音频未到齐 / 尚未推理
        REVISABLE,   // 已预测，但后续可修改
        STABLE       // 已提交，不再修改
    };

    // 模型输出
    float emotion_logits[C];
    float level_logits[K];
    float boundary_prob;

    // 解码后结果
    int emotion_id;
    int level_id;
    bool is_boundary;

    // 置信度/版本
    float confidence;
    int revision_id;      // 这帧被改写过多少次
    int model_pass_id;    // 来自哪次推理批次

    // 时间信息
    double t_sec;
}


⸻

Buffer 不是环形“裸数组”，而是“时间轴表 + 滑动窗口”

建议维护三个逻辑区域：

1. 历史 stable 区
已经提交给播放器的帧
不可再写，只读或可归档

2. 活跃 revise 区
最近几秒的帧，可被模型覆盖重写

3. 未到达区
未来帧，尚不存在或占位

⸻

推荐整体缓存长度

你不需要无限缓存全历史。
建议维护：

history_keep = 20s ~ 60s
revise_window_keep = 8s ~ 12s

例如：
	•	stable 历史保留最近 30 秒
	•	revisable 区始终保留最近 10 秒
	•	更早数据归档或压缩

⸻

写入规则

对 REVISABLE 帧
允许模型覆盖：
	•	logits
	•	decoded labels
	•	boundary
	•	confidence

并且：

revision_id += 1

对 STABLE 帧
默认禁止覆盖。
除非你设计了“极端回滚机制”，但第一版别做，复杂且容易和播放器打架。

⸻

1.6 什么时候把 REVISABLE 变成 STABLE

每次推理后，计算：

stable_end = T_ingest - R_stable

然后把所有：

status == REVISABLE and frame_idx <= stable_end

的帧升级成 STABLE。

注意，不是“新推理到哪儿就 stable 到哪儿”，而是要按右上下文规则推进。

⸻

1.7 边界的 commit 规则要比普通帧更严格

边界是离散事件，比 frame emotion 更敏感。
建议边界用双层提交：

层 1：frame stable

某帧 emotion / level 已经稳定

层 2：boundary stable

边界点还要满足：
	•	峰值位置不再漂移
	•	左右最小段长满足
	•	邻近更强峰未出现

所以边界可以单独再加一个确认条件：

boundary stable if:
    frame stable
    and peak is older than R_boundary_confirm

建议：

R_boundary_confirm = 9 ~ 15 frames

即 300~500ms。

⸻

1.8 三个播放器怎么取数

你这里最容易出事故的点，就是三个播放器各自“理解时间”的方式不同。
所以一定要约定：

三个播放器都只认同一主轴 frame_idx，
不各自算时间、不各自做平滑、不各自猜状态。

⸻

播放器 A：主音频播放器

它的真实播放时钟定义了：

play_frame_idx = floor((now - playback_start_ts) * fps)

它本身不需要 emotion 逐帧参与声音播放，但需要同步状态给 UI / 上层行为逻辑。

它读取：

timeline[play_frame_idx]

读取策略：
	•	优先取 STABLE
	•	若没有 STABLE，可临时取 REVISABLE
	•	若还没有，回退到最近 stable 状态或 neutral

⸻

播放器 B：帧级节奏播放器

这个要求最高，必须帧对齐。

它应该严格按：

frame_idx -> FrameState

读取，不做内部插值。

建议读取：
	•	emotion_id
	•	level_id
	•	is_boundary

如果节奏播放器特别吃边界，可以给它单独提供：

segment_phase
boundary_strength

但来源仍然必须是统一 timeline。

⸻

播放器 C：第三播放器 / 上层控制播放器

如果它需要的是“当前情绪态”，不一定要消费每帧原始 logits，而是应该消费一个更稳的接口：

CurrentEmotionState {
    emotion_id
    level_id
    confidence
    segment_id
    is_boundary_on_this_frame
}

也就是从统一 timeline 派生，而不是自己二次推理。

⸻

1.9 建议的线程与模块划分

线程 1：Ingress Thread

负责：
	•	接收 TTS PCM 流
	•	写入 PCM ring buffer
	•	更新 T_ingest

线程 2：Feature Thread

负责：
	•	把新增 PCM 转成 mel frame
	•	写入 feature buffer

线程 3：Inference Thread

每个 infer_tick 触发：
	•	读取最近 W_left_recompute 到 T_ingest
	•	跑模型
	•	覆盖 revise 区
	•	推进 stable 边界

线程 4：Playback Sync Thread

负责：
	•	根据主音频播放时钟得到 play_frame_idx
	•	把当前帧结果分发给三个播放器

⸻

1.10 建议的系统协议量化版本

给你一个可以直接开工的初始版本：

sample_rate           = 16000
fps                   = 30
frame_hop             = 533 samples

二、模型张量规格

下面把模型定义到“可以开始写 module”的粒度。

⸻

2.1 音频帧率

建议继续沿用你的现有体系：

fps = 30
frame_hop_sec = 1/30 ≈ 33.3ms

优点前面说过，不重复。

⸻

2.2 每次模型看到的音频张量

假设一次推理针对最近一个窗口。
窗口长度：

W_total = W_left_context + W_current + W_right_available

在你的系统里，其实输入到模型的是：

最近一段已到达音频对应的 mel 帧

推荐第一版直接设最大输入窗口：

T_max = 180 frames  (~6s)

对应张量：

audio_mel: [B, T, 80]

其中：
	•	B = 1（在线单流）
	•	T <= 180
	•	80 是 mel bins

⸻

2.3 lookahead 帧数

这是最关键的模型参数。

注意这里有两个概念：

1. R_model

模型编码单帧所需右上下文

2. R_stable

系统提交单帧所需右上下文

模型本身建议先取：

lookahead_model = 24 frames ≈ 0.8s

这是个很好的平衡点。

如果用 30fps：
	•	12 帧 = 0.4s，偏小
	•	24 帧 = 0.8s，比较稳
	•	36 帧 = 1.2s，更稳但延迟更高

我建议第一版：

lookahead_model = 24


⸻

2.4 文本长度上限

因为文本一次输入，且一般是 TTS 输入文本，不会无限长。
但你要防御性设计。

建议两级限制：

方案 A：token 上限

L_text_max = 128

对绝大多数单句、多子句 TTS 足够。

方案 B：更宽松

L_text_max = 192 or 256

如果你常见文本更长，可以上 192。

⸻

我建议

第一版：

L_text_max = 128

超过时：
	•	截断
	•	或分句后只取当前 TTS 对应文本单元

⸻

2.5 Text Encoder 输出张量

text_tokens       : [B, L]
text_attention_ms : [B, L]
text_hidden       : [B, L, D]

推荐：

D = 256

不要第一版就上 768 全尺寸，在线工程没必要太重。

⸻

2.6 Audio Encoder 输出张量

audio_hidden: [B, T, D]

同样：

D = 256

这样 audio/text 维度一致，方便 cross-attention。

⸻

2.7 cross-attention 放在哪

这是个重要设计点。

我建议不是一开始就融合，也不是最后才融合，而是：

Audio 先做 24 层单模态编码，再做 cross-attention，再做 24 层 joint 编码。

即三段式：

Stage 1：Audio local encoding

学习基础韵律/能量/停顿/声学情绪

Stage 2：Audio→Text cross-attention

把全文文本语义注入到每帧

Stage 3：Joint temporal refinement

再用几层时序层把融合结果沿时间轴整理

⸻

推荐结构

Input audio_mel [B,T,80]
  ↓
Audio frontend (linear / conv)
  ↓
Audio encoder block × 3
  ↓
audio_hidden_1 [B,T,256]

Text tokens [B,L]
  ↓
Text encoder block / pretrained LM
  ↓
text_hidden [B,L,256]

Cross-attn block × 2
  Q = audio_hidden_1
  K,V = text_hidden
  ↓
audio_text_hidden [B,T,256]

Joint temporal block × 2
  ↓
joint_hidden [B,T,256]

这就是很适合你的第一版结构。

⸻

2.8 三个 head 的输出形状

你当前文档是：
	•	type_logits [T,7]
	•	lvl_logits [T,6]
	•	bnd_logits [T]  ￼

端到端版我建议略微调整：

⸻

Head 1：emotion head

建议类别数：

C = 7

定义为：
	•	happy
	•	sad
	•	angry
	•	fear
	•	calm
	•	confused
	•	neutral

注意：我建议把你原文档里 reserved 的那个类改成 neutral，这对在线稳定性很重要。
否则所有无明显情绪的帧会被硬挤到别的类里。 ￼

输出：

emotion_logits: [B, T, 7]
emotion_prob  : [B, T, 7]
emotion_id    : [B, T]


⸻

Head 2：level head

保留 6 级没问题：

K = 6

输出：

level_logits: [B, T, 6]
level_prob  : [B, T, 6]
level_id    : [B, T]

如果后续想更平滑，也可以改成回归标量 [0,1]，但第一版先分类即可。

⸻

Head 3：boundary head

输出：

boundary_logit: [B, T]
boundary_prob : [B, T]

这是逐帧概率。

但建议模型内部再额外输出一个辅助量：

boundary_peakness: [B, T]   (optional)

第一版可以没有。
有它的话，后续边界解码更容易稳。

⸻

2.9 可选第四个 head：stability / confidence

这个不是必须，但对你的系统很有价值。

因为你有 REVISABLE -> STABLE 这个生命周期，
模型如果能额外估计“这一帧现在有多稳”，系统会更聪明。

可选输出：

stability_logit: [B, T]
stability_prob : [B, T]

它的含义可定义为：

在当前前视条件下，这一帧预测未来是否还会变。

第一版不做也可以，先用固定 R_stable 规则推进。

⸻

2.10 推荐的具体张量规格汇总

给你一个可直接实现的版本：

Audio

audio_pcm_chunk         : [N]                  // 增量输入
audio_mel_window        : [1, T, 80]          // T <= 180
audio_hidden            : [1, T, 256]

Text

text_token_ids          : [1, L]              // L <= 128
text_attention_mask     : [1, L]
text_hidden             : [1, L, 256]

Fusion

cross_attn_output       : [1, T, 256]
joint_hidden            : [1, T, 256]

Heads

emotion_logits          : [1, T, 7]
level_logits            : [1, T, 6]
boundary_logits         : [1, T]

Decoded

boundary_prob           : [T]

三、把系统时序和模型张量对上

你现在真正要的是“模型输出如何变成播放器可消费的 timeline”。

可以这样对齐：

⸻

3.1 模型输出写回 timeline buffer

假设本次重算覆盖：

[t0, t1]

模型输出 T = t1 - t0 + 1 帧结果。

那么写回规则是：

timeline[t0 + i].emotion_logits = emotion_logits[i]
timeline[t0 + i].level_logits   = level_logits[i]
timeline[t0 + i].boundary_prob  = boundary_prob[i]
timeline[t0 + i].status         = REVISABLE or STABLE
timeline[t0 + i].revision_id   += 1


⸻

3.2 stable 推进

本次 ingest 最远是 T_ingest，则：

stable_end = T_ingest - 42

将：

frame_idx <= stable_end

的 revisable 帧升级为 stable。

⸻

3.3 播放线程消费

当前播放到：

play_idx

则三个播放器统一读取：

timeline[play_idx]

如果你担心偶发未准备好，则定义 fallback：

优先级
	1.	STABLE
	2.	REVISABLE
	3.	最近 stable 帧
	4.	neutral 默认值

⸻

四、如果你已经能采集真实输入流格式，下一步最该做什么

你现在最有价值的，不是立刻训练大模型，而是先用真实流把协议跑通。

我建议按这三个阶段推进：

⸻

阶段 1：只做系统壳，不接新模型

先用你现有规则/模型，模拟新协议：
	•	接真实 PCM 流
	•	建 timeline buffer
	•	做 revisable/stable 切换
	•	三播放器统一按 frame_idx 取数

这一步先验证：
	•	时间轴对不对
	•	稳定区推进是否永远领先播放
	•	边界 commit 是否平滑

⸻

阶段 2：换新模型，但保留后处理

把 backbone 换成多模态 Transformer，
但边界解码、segment voting、stable 规则先保留。

⸻

阶段 3：再逐步学习化

把一部分手工规则移进模型。

⸻

五、我建议你现在补采的真实流字段

既然你说有工程可以采真实输入流格式，那建议你下一步把这些字段采出来。
这会极大提升后面设计精度：

音频流侧
	•	每个 chunk 到达时间戳
	•	chunk sample 数
	•	chunk 序号
	•	累计到达 sample 数

播放侧
	•	主音频播放开始时间
	•	当前播放时间戳
	•	当前播放 sample/frame
	•	是否有 pause / seek / drift

TTS 侧
	•	首包时间
	•	尾包时间
	•	平均传入速度相对播放速度的倍率

文本侧
	•	一次输入全文
	•	文本长度
	•	token 数分布
	•	是否有多语言 / 标点密度

现有情绪系统侧
	•	现有输出帧率
	•	边界事件延迟
	•	输出抖动情况
