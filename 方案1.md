好，下面我直接按你要的形式，给一版更像正式设计文档的内容。先只讲系统时序 / buffer / 三播放器协议，默认第一阶段不换模型，沿用你现有的 TCN、boundary head、后处理链路。你现有系统已经有帧级 type / level / boundary 输出、边界事件、段内投票、延迟发射这些要素，所以这一版的核心不是重做算法，而是把它们放进统一协议里。 ￼

⸻

Speech2Emotion 在线系统协议设计文档（Phase 1：不换模型）

1. 设计目标

第一阶段目标只有一个：

先把“模型输出”变成“播放器可稳定消费的统一时间轴对象”。

当前主要问题不是分类头本身，而是：
	•	文本与音频仍是伪对齐
	•	三个播放器没有共享同一时间协议
	•	输出没有区分“暂定结果”和“最终结果”
	•	播放层直接消费模型瞬时输出，导致语义不稳定

因此第一阶段只解决四件事：
	1.	统一 frame_idx 主轴
	2.	引入 commit / revise 生命周期
	3.	明确三个播放器的读取语义
	4.	用动态前视规则推进提交线

⸻

2. 系统总览

系统中同时存在 4 条时间相关链路：
	1.	TTS 音频传入链路
定义 ingest_time / ingest_frame
	2.	主音频播放链路
定义 playback_time / play_frame
	3.	模型推理链路
定义 infer_frame
	4.	情绪输出时间轴
定义 timeline[frame_idx]

这四者关系是：
	•	音频先被传入
	•	模型基于已传入部分增量推理
	•	timeline 被不断写入和修正
	•	播放器按主音频播放时钟读取 timeline

所以这是一个：

输入增量到达、结果可修正、消费按播放主轴进行的时序系统

⸻

3. 统一时间协议

3.1 唯一主轴

全系统共享唯一离散主轴：

frame\_idx \in \mathbb{N}

推荐直接沿用你当前 30fps 帧轴，因为现有 Mel 特征和流式推理本来就是按约 1/30 s 的帧移组织的。 ￼

定义：

fps = 30

t_{sec}(f) = f / fps

f(t) = \lfloor t \cdot fps \rfloor

所有外部时间都必须先投影到 frame_idx，再参与协议。

⸻

3.2 三个时间量

3.2.1 播放时间

F_{play}(t)

表示当前真实播放到的帧。
由主音频播放器定义，是整个系统的最终消费时钟。

⸻

3.2.2 传入时间

F_{ingest}(t)

表示当前已经收到的最远音频帧。
由TTS 音频接收线程维护。

⸻

3.2.3 推理时间

F_{infer}(t)

表示当前模型已经写入 timeline 的最远帧。
由推理线程维护。

⸻

3.3 提交时间

再定义一个关键量：

F_{commit}(t)

表示当前已可视为最终结果的最远帧。
这是播放器真正应该优先消费的边界。

⸻

3.4 四条线的关系

一般情况下有：

F_{play}(t) \le F_{commit}(t) \le F_{infer}(t) \le F_{ingest}(t)

但系统并不保证始终成立。
一旦 commit 追不上 play，就会触发 fallback。

⸻

4. 帧生命周期协议

4.1 状态枚举

每个帧必须有明确状态：

enum class FrameStatus : uint8_t {
    UNSEEN = 0,       // 音频尚未到达
    UNREADY = 1,      // 音频已到达，但尚未推理
    PROVISIONAL = 2,  // 已有预测，但允许后续改写
    COMMITTED = 3     // 已提交，播放器默认视为最终结果
};


⸻

4.2 各状态定义

UNSEEN
	•	该帧对应音频还没到
	•	timeline 可有占位，但不可读

UNREADY
	•	音频已到，但模型还没覆盖该帧
	•	说明接收线程快于推理线程

PROVISIONAL
	•	已有模型输出
	•	允许因未来上下文到来而被重写
	•	可供内部预览或应急 fallback 使用

COMMITTED
	•	已获得足够右上下文与稳定性保证
	•	第一阶段默认不再修改
	•	播放器应优先读取它

⸻

4.3 committed 的严格定义

帧 f 成为 COMMITTED 的条件不是“已经推理过”，而是：
	1.	它已经被推理覆盖
	2.	它拥有足够右侧未来上下文
	3.	它经过当前后处理链路后可被视为最终结果

形式化写成：

f \le F_{infer}(t)
且
F_{ingest}(t) - f \ge R_{stable}(t)

于是：

F_{commit}(t) = \min(F_{infer}(t),\, F_{ingest}(t) - R_{stable}(t))

凡满足：

f \le F_{commit}(t)

的帧，可升级为 COMMITTED。

⸻

5. 动态前视窗口公式

这是协议核心。

5.1 当前可用前视

定义相对当前播放位置的可用未来帧数：

L(t) = F_{ingest}(t) - F_{play}(t)

含义：
	•	L(t)=0：无未来信息，退化为严格在线
	•	L(t)>0：系统可看到部分未来
	•	L(t) 越大，播放附近帧越容易稳定提交

⸻

5.2 可消费前视

仅音频到达还不够，推理也得跟上，所以再定义：

L_{infer}(t) = F_{infer}(t) - F_{play}(t)

以及：

L_{commit}(t) = F_{commit}(t) - F_{play}(t)

真正播放器可稳定消费的是 L_commit(t)。

⸻

5.3 stable 条件

一个帧 f 在时刻 t 稳定的基本条件：

F_{ingest}(t) - f \ge R_{stable}(t)

工程上用提交线表达更方便：

f \le F_{commit}(t)

⸻

5.4 R_stable 的拆分

建议拆成：

R_{stable} = R_{model} + R_{boundary} + R_{margin}

R_model

模型为使该帧判定稳定所需的右上下文帧数
第一阶段如果仍用现有 TCN，可把它看成“经验稳定窗”，不是严格网络结构参数

R_boundary

边界事件确认需要的额外未来帧
因为边界往往比普通 emotion 分类更敏感

R_margin

工程安全裕量，覆盖：
	•	infer tick 周期
	•	单次推理耗时
	•	调度抖动
	•	buffer 写回延迟

⸻

5.5 stable_right_frames 如何由真实速度推出来

设：
	•	fps：帧率
	•	Δ_tick：推理触发周期
	•	Δ_inf：单次推理 + 写回耗时
	•	Δ_jitter：调度抖动裕量
	•	R_model：模型右侧稳定需求
	•	R_boundary：边界确认需求

则推荐：

R_{margin} = \lceil fps \cdot (\Delta_{tick} + \Delta_{inf} + \Delta_{jitter}) \rceil

于是：

R_{stable} = R_{model} + R_{boundary} + R_{margin}

例子：
	•	fps = 30
	•	Δ_tick = 0.30s
	•	Δ_inf = 0.15s
	•	Δ_jitter = 0.10s

则：

R_{margin} \approx 30 \cdot 0.55 = 16.5 \approx 17

若经验上：
	•	R_model = 18
	•	R_boundary = 12

则：

R_{stable} = 47

也就是约 1.57 秒。

第一阶段完全可以先取固定值，例如 42~48 帧，后续再用真实日志反推。

⸻

5.6 何时掉到 provisional fallback

播放器在播放 F_play(t) 时，如果：

F_{commit}(t) < F_{play}(t)

说明提交线落后于播放线，读不到 committed。

这时分两种情况：

A. 可读 provisional

若：

F_{infer}(t) \ge F_{play}(t)

说明至少该帧已被推理覆盖，可读 PROVISIONAL。

B. 连 provisional 都没有

若：

F_{infer}(t) < F_{play}(t)

说明推理线也落后播放线，只能回退到历史状态或默认值。

⸻

6. commit / revise buffer 设计

这一部分按接近实现的粒度给。

6.1 每帧数据结构

struct FrameRecord {
    int frame_idx = -1;
    double t_sec = 0.0;

    // 生命周期
    FrameStatus status = FrameStatus::UNSEEN;

    // 到达 / 推理标记
    bool audio_arrived = false;
    bool inferred = false;

    // 原始输出
    float emotion_logits[7] = {0};   // 沿用现有 type 头，7 类
    float level_logits[6]   = {0};   // 现有 level 头，6 类
    float boundary_prob     = 0.0f;

    // 解码结果
    int emotion_id = -1;
    int level_id   = -1;
    bool boundary_flag = false;

    // 置信度
    float emotion_conf = 0.0f;
    float level_conf   = 0.0f;
    float final_conf   = 0.0f;

    // 所属段
    int segment_id = -1;

    // 修订跟踪
    uint32_t revision_id = 0;
    uint32_t infer_pass_id = 0;

    // 诊断字段
    int source_begin_frame = -1;
    int source_end_frame   = -1;
};


⸻

6.2 revision_id 是否需要

需要，而且第一阶段就应该有。

原因：
	•	能看到哪些帧在 commit 前被频繁改写
	•	能看到哪个区间一直不稳定
	•	能判断当前效果差是否来自协议层波动
	•	后续调 R_stable 时能直接用

这个字段很值钱。

⸻

6.3 boundary：帧属性还是独立事件流

结论：两者都要

放在帧上

每帧都有：
	•	boundary_prob
	•	boundary_flag

因为节奏播放器是逐帧消费的。

额外保留独立事件流

因为边界本质上又是离散事件，便于段表维护：

enum class EventStatus : uint8_t {
    PROVISIONAL = 0,
    COMMITTED = 1
};

struct BoundaryEvent {
    int boundary_id = -1;
    int frame_idx = -1;
    double t_sec = 0.0;

    float confidence = 0.0f;
    EventStatus status = EventStatus::PROVISIONAL;

    int left_segment_id = -1;
    int right_segment_id = -1;

    uint32_t revision_id = 0;
};


⸻

6.4 segment state 放在哪里

建议：
	•	帧上只放 segment_id
	•	完整段状态放段表里

不要把完整段统计重复塞到每帧上。

struct SegmentState {
    int segment_id = -1;

    int start_frame = -1;
    int end_frame = -1;
    bool end_closed = false;

    EventStatus status = EventStatus::PROVISIONAL;

    int major_emotion_id = -1;
    int major_level_id = -1;

    int emotion_hist[7] = {0};
    int level_hist[6]   = {0};

    int frame_count = 0;
    float confidence = 0.0f;
};


⸻

6.5 timeline buffer 容器

struct TimelineBuffer {
    std::deque<FrameRecord> frames;

    int base_frame_idx = 0;      // frames.front() 对应的真实 frame_idx
    int ingest_end = -1;         // 最远已到达帧
    int inferred_end = -1;       // 最远已推理帧
    int committed_end = -1;      // 最远已提交帧
};

另外维护：

std::deque<BoundaryEvent> boundary_events;
std::deque<SegmentState>  segments;


⸻

6.6 历史 stable 区和尾部 revisable 区

逻辑上分三段：

A. committed 历史区

[base\_frame\_idx,\ committed\_end]

B. provisional 活跃区

(committed\_end,\ inferred\_end]

C. unready 区

(inferred\_end,\ ingest\_end]

再往后是 unseen。

第一阶段建议不拆成多个物理 buffer，逻辑区分就够了。

⸻

6.7 buffer 裁剪

建议：
	•	committed 历史保留 20~30 秒
	•	活跃 revisable 区保留 8~12 秒

例如 30fps 下：
	•	history_keep_frames = 900
	•	revise_keep_frames = 300

裁剪策略：
	•	只裁掉很旧的 committed
	•	不裁剪仍可能被回写的 provisional 尾部

⸻

7. 三播放器协议

下面分播放器定语义。

⸻

7.1 总原则

三个播放器都共享同一个 frame_idx 主轴。
但它们不必直接裸读 timeline[frame_idx]。

推荐做法：
	•	timeline 是统一底层真值
	•	对不同播放器导出不同视图对象

这样既统一协议，又避免每个消费方自己二次解释。

⸻

7.2 主音频播放器

它读什么

主音频播放器不需要原始 logits，它要的是“当前播放帧的情绪视图”：

struct PlaybackEmotionView {
    int frame_idx = -1;
    FrameStatus status = FrameStatus::UNSEEN;

    int emotion_id = -1;
    int level_id = -1;
    float confidence = 0.0f;

    int segment_id = -1;
    bool is_boundary = false;
};

它的职责
	•	提供真实播放时钟
	•	同步当前播放位置的情绪状态给 UI / 状态层

是否直接读 timeline

不建议。应通过：

PlaybackEmotionView get_playback_view(int play_frame_idx);


⸻

7.3 帧级节奏播放器

它是最严格的逐帧消费方。

它读什么

struct RhythmFrameView {
    int frame_idx = -1;
    FrameStatus status = FrameStatus::UNSEEN;

    int emotion_id = -1;
    int level_id = -1;

    bool boundary_flag = false;
    float boundary_strength = 0.0f;
};

它关心什么
	•	当前帧 emotion
	•	当前帧 level
	•	当前帧是否是边界

是否直接读 timeline

可以非常接近 timeline，但仍建议通过接口：

RhythmFrameView get_rhythm_view(int frame_idx);

避免它自己做状态推断。

⸻

7.4 上层控制 / 表现层播放器

这个消费方通常不要逐帧原始细节，而要更稳的“状态摘要”。

它读什么

struct ControlStateView {
    int frame_idx = -1;
    FrameStatus status = FrameStatus::UNSEEN;

    int emotion_id = -1;
    int level_id = -1;
    float confidence = 0.0f;

    int segment_id = -1;
    int segment_major_emotion = -1;
    int segment_major_level = -1;
    float segment_progress = 0.0f;

    bool is_boundary_on_this_frame = false;
};

它是否直接读 timeline

不建议。
它应由 timeline + segment table + boundary events 联合派生：

ControlStateView get_control_view(int frame_idx);


⸻

8. fallback 策略

这是最容易让系统看起来“抽搐”的部分，必须定死。

8.1 主音频播放器 fallback

优先级：
	1.	当前帧 COMMITTED
	2.	当前帧 PROVISIONAL
	3.	最近一个 COMMITTED 帧的状态
	4.	默认 neutral/calm

原因：它更偏“同步当前状态”，允许短暂 provisional。

⸻

8.2 帧级节奏播放器 fallback

优先级更保守：
	1.	当前帧 COMMITTED
	2.	最近一个 COMMITTED 帧的稳定状态
	3.	默认 neutral/calm，且 boundary=false

原因：节奏播放器最怕抖动，不应轻易吃 provisional 跳变。

⸻

8.3 上层控制 / 表现层 fallback

优先级：
	1.	当前帧的 committed control view
	2.	当前 segment 的最近 committed 摘要
	3.	上一个 committed segment 的持续状态
	4.	默认安全状态

原因：表现层最怕来回闪烁，宁可慢半拍，也不要抖。

⸻

9. 读取策略：哪些量给谁

量	主音频播放器	帧级节奏播放器	上层控制/表现层
emotion_id	要	要	要
level_id	要	要	要
boundary_flag	要	强依赖	要
boundary_prob	可选	要	一般不要直接暴露
segment_id	可选	可选	要
segment_major_emotion	一般不必	一般不必	强依赖
current emotion state	要	不单独要	要

结论：
	•	boundary：帧级节奏播放器最依赖
	•	segment major：上层控制/表现层最依赖
	•	当前 emotion state：主音频播放器和控制层都要

⸻

10. 第一阶段最小落地路线（不换模型）

这里完全按你的要求，只动协议层。

10.1 保留不动

全部保留：
	•	当前 TCN
	•	当前 boundary head
	•	当前后处理
	•	当前文本 prior / 规则层
	•	当前伪对齐方式

也就是说，先承认模型不是当前第一瓶颈。

⸻

10.2 只新增四件事

1. 统一 frame_idx

所有输出统一落在同一帧轴

2. timeline buffer

把当前输出写进 FrameRecord

3. commit / revise 生命周期

引入 PROVISIONAL / COMMITTED

4. 动态 lookahead 提交

按 F_commit = min(F_infer, F_ingest - R_stable) 推进

⸻

10.3 最小流程

Step A：音频接收
	•	TTS chunk 到来
	•	更新 PCM / feature buffer
	•	推进 F_ingest
	•	新到达帧标记为 UNREADY

Step B：推理 tick
	•	每隔固定 tick 取最近一个重算窗口
	•	用当前 TCN + 后处理跑一遍
	•	写回这些帧为 PROVISIONAL
	•	推进 F_infer

Step C：提交推进
	•	根据 F_ingest 与 R_stable 算出 F_commit
	•	将 <= F_commit 的 provisional 帧改为 committed
	•	同步提交对应 boundary / segment

Step D：播放读取
	•	三播放器统一按 frame_idx 读 view
	•	不再直接各自消费模型瞬时输出

⸻

10.4 第一阶段要验证什么

不是先看分类指标，而是先看系统行为：

1. 对齐是否改善

三个播放器是否终于看到同一时间点的同一状态

2. 消费是否顺了

主音频、节奏、控制层是否不再互相错位

3. 抖动是否下降

commit/revise 后，是否明显减少“刚出来就变”的不稳定输出

4. 当前差效果是否主要卡在协议层

如果协议一理顺，体验显著提升，就说明你现在的判断基本正确

⸻

11. 推荐第一版参数

在没打实测日志前，先给一版工程初值：

fps                   = 30
audio_chunk_ms        = 200 ~ 300
infer_tick_ms         = 250 ~ 350
recompute_left_frames = 90 ~ 120

R_model               = 18
R_boundary            = 12
R_margin              = 12
R_stable              = 42

history_keep_frames   = 900   // 30s
revise_keep_frames    = 300   // 10s

其中 R_stable = 42 对应约 1.4 秒，是一个比较稳的起点。

⸻

12. 这阶段必须补的日志

为了把参数从“经验值”变成“协议值”，建议立即加这几类日志：

12.1 时间线日志
	•	F_play
	•	F_ingest
	•	F_infer
	•	F_commit

12.2 fallback 日志
	•	当前帧读到的是 committed / provisional / history / default 哪种

12.3 revision 日志
	•	每帧 revision_id
	•	commit 前被改写次数的分布

12.4 boundary 日志
	•	边界首次出现时刻
	•	边界最终 commit 时刻
	•	边界位置漂移量

有了这批日志，你就能回答最关键的问题：

现在效果差，到底是模型不行，还是 timeline 协议没立住。

⸻

13. 第一阶段之后，再进入第二阶段

按你现在的诉求，第二阶段才该进入模型层，顺序我建议固定为：
	1.	先把系统协议跑通
	2.	再上 bounded right-context audio model
	3.	再把文本作为 memory 接入
	4.	最后再逐步用 learned fusion 替换 text prior / 硬规则

也就是：

先解决“输出怎么成为系统对象”，再解决“模型怎么变强”。

⸻

14. 这份设计的核心结论

一句话收敛：

你当前第一阶段最应该做的，不是换大模型，而是先把每一帧从“瞬时推理结果”升级成“带生命周期的 timeline 对象”，再让三个播放器只通过统一 frame 主轴和 commit/revise 语义去消费它。

15.4 样本数到 frame_idx 的换算必须与当前特征器严格一致

这一点在当前代码里其实已经定死了，不能口算近似。

根据 `models/featurizer.py`：
	•	`hop_length = round(16000 * 1/30) = 533`
	•	`n_fft = round(16000 * 0.05) = 800`
	•	`center = False`

因此，在 Phase 1 的 ingest 线程里：

若累计收到样本数为 `N_samples`，则最远可提特征帧数为：

\[
T_{ready} =
\begin{cases}
0, & N_{samples} < n\_fft \\
1 + \left\lfloor \frac{N_{samples} - n\_fft}{hop\_length} \right\rfloor, & N_{samples} \ge n\_fft
\end{cases}
\]

于是最远已到达帧：

\[
F_{ingest} = T_{ready} - 1
\]

代入当前仓库参数即：
	•	`hop_length = 533`
	•	`n_fft = 800`

这一条要直接写死在运行时实现里，不要让不同线程各自猜。

16.1 `update_ingest()`

职责：
	•	接收最新 PCM chunk
	•	更新样本计数与 `F_ingest`
	•	为新到达帧分配 `FrameRecord`
	•	把这些帧标成 `UNREADY`

伪代码：

```cpp
void update_ingest(const float* pcm, int n_samples, double recv_ts_sec) {
    pcm_ring.append(pcm, n_samples);
    total_samples_received += n_samples;

    int new_ingest_end = samples_to_ready_last_frame(
        total_samples_received,
        /*n_fft=*/800,
        /*hop_length=*/533
    );

    if (new_ingest_end <= timeline.ingest_end) {
        return;
    }

    for (int f = timeline.ingest_end + 1; f <= new_ingest_end; ++f) {
        FrameRecord rec;
        rec.frame_idx = f;
        rec.t_sec = double(f) / 30.0;
        rec.audio_arrived = true;
        rec.status = FrameStatus::UNREADY;
        timeline.frames.push_back(rec);
    }

    timeline.ingest_end = new_ingest_end;
}
```

实现注意：
	•	这一步绝不做推理
	•	只负责推进“音频物理到达线”

⸻

16.2 `run_infer_tick()`

职责：
	•	每隔固定 tick 触发一次
	•	从 PCM ring / feature buffer 里截取一个“重算窗口”
	•	调用现有声学模型
	•	调用现有文本融合与情绪约束
	•	把结果写回 timeline 为 `PROVISIONAL`

Phase 1 推荐窗口定义：

\[
W_{ctx\_begin} = \max(base\_frame,\ committed\_end + 1 - W_{left})
\]

\[
W_{write\_begin} = committed\_end + 1
\]

\[
W_{end} = ingest\_end
\]

含义：
	•	`W_ctx_begin`：提供给当前 TCN 的左上下文
	•	`W_write_begin`：真正允许被改写的首帧
	•	`W_end`：当前已到达的最远帧

伪代码：

```cpp
void run_infer_tick(double now_sec) {
    if (timeline.ingest_end < 0) return;

    int write_begin = std::max(timeline.base_frame_idx, timeline.committed_end + 1);
    int ctx_begin   = std::max(timeline.base_frame_idx, write_begin - recompute_left_frames);
    int write_end   = timeline.ingest_end;

    if (write_begin > write_end) return;

    WaveSlice wav = pcm_ring.slice_frames(
        ctx_begin,
        write_end,
        /*n_fft=*/800,
        /*hop_length=*/533
    );

    AcousticSlice acoustic = acoustic_adapter.infer_window(
        wav,
        /*ctx_begin_frame=*/ctx_begin,
        /*write_begin_frame=*/write_begin,
        /*write_end_frame=*/write_end
    );

    FusedSlice fused = fusion_adapter.refine_window(
        acoustic,
        committed_boundary_events,
        committed_segments,
        full_text
    );

    for (auto& fr : fused.frames) {
        FrameRecord& dst = timeline.at(fr.frame_idx);
        dst.inferred = true;
        dst.status = FrameStatus::PROVISIONAL;
        dst.revision_id += 1;
        dst.infer_pass_id = infer_pass_id;

        dst.emotion_id = fr.emotion_id;
        dst.level_id = fr.level_id;
        dst.boundary_prob = fr.boundary_prob;
        dst.boundary_flag = fr.boundary_flag;
        dst.segment_id = fr.segment_id;
        dst.emotion_conf = fr.emotion_conf;
        dst.level_conf = fr.level_conf;
        dst.final_conf = fr.final_conf;
        dst.source_begin_frame = ctx_begin;
        dst.source_end_frame = write_end;
    }

    overwrite_provisional_events_and_segments(fused);
    timeline.inferred_end = std::max(timeline.inferred_end, write_end);
    infer_pass_id += 1;
}
```

16.3 `advance_commit_line()`

职责：
	•	根据 `F_ingest`、`F_infer` 和 `R_stable` 计算新的 `F_commit`
	•	把满足条件的帧从 `PROVISIONAL` 升级成 `COMMITTED`
	•	提交对应的 boundary event 和 segment

伪代码：

```cpp
void advance_commit_line() {
    int new_commit_end = std::min(
        timeline.inferred_end,
        timeline.ingest_end - stable_right_frames
    );

    if (new_commit_end <= timeline.committed_end) {
        return;
    }

    for (int f = timeline.committed_end + 1; f <= new_commit_end; ++f) {
        FrameRecord& rec = timeline.at(f);
        if (rec.status == FrameStatus::PROVISIONAL) {
            rec.status = FrameStatus::COMMITTED;
        }
    }

    promote_boundary_events_to_committed(new_commit_end);
    close_or_promote_segments(new_commit_end);

    timeline.committed_end = new_commit_end;
}
```

关键原则：
	•	帧提交线推进后，才允许 boundary / segment 进入 committed
	•	提交是单调向前的，第一阶段不做回滚

17. 边界与段的提交细则（结合当前代码补齐）

这一节是当前 `方案1.md` 里还差半步的地方，也是最容易在落地时踩坑的地方。

17.1 为什么边界不能只跟帧一起提交

因为你当前代码里：
	•	帧是逐帧存在的
	•	边界本质是峰值事件
	•	segment 又依赖边界来闭合

所以不能只说“帧 committed 了，边界自然 committed”。
要明确三层关系：
	1.	帧级 `boundary_prob`
	2.	事件级 `BoundaryEvent`
	3.	段级 `SegmentState`

⸻

17.2 Phase 1 的边界提交规则

建议第一阶段用最保守的规则：

边界事件 `b` 变成 `COMMITTED` 的条件：
	1.	`b.frame_idx <= timeline.committed_end`
	2.	该边界在最近一次 provisional 重算中仍存在
	3.	其左右相邻边界关系未违反 `min_gap`

换句话说：
	•	帧级提交线是前提
	•	边界事件还要在最新 active window 内“存活下来”

不要求第一阶段就支持：
	•	边界提交后回滚
	•	已提交边界重新漂移

⸻

17.3 segment 的提交规则

建议按“右边界提交后段才闭合”的原则：

对于一个段 `seg_k`：
	•	左边界 committed 后，它可以存在
	•	但 `end_closed=false`
	•	只有当右边界 committed 后：
		•	`end_frame` 才可确定
		•	`major_emotion` / `major_level` 才升级为 committed

这和你当前 `stream_online_emotion.py` 中
“先有 boundary event，再重建 segment major”
的逻辑是一致的，只是现在把它拆成了生命周期协议。

⸻

17.4 当前文本先验在 provisional 阶段不能污染全局状态

这是结合你现有代码必须补上的关键规则。

因为在 `online_emotion/detector.py` 里，当前 `OnlineBoundaryDetector._emit_event()` 会立刻：

	•	生成 `BoundaryEvent`
	•	更新 `last_confirmed`
	•	调用 `text_prior.register_boundary()`

也就是说：
当前 detector 的确认事件会直接修改文本先验映射。

如果把这件事直接用于 provisional 区，就会出大问题：
	•	今天 provisional 边界 A 出现了
	•	它改写了 `TextPrior`
	•	下一轮重算 A 消失了
	•	但全局 prior 已被污染

所以 Phase 1 必须明确：

只有 COMMITTED boundary event 才能写回全局 `TextPrior` 状态。

provisional 阶段应使用：
	•	全局 committed prior 的快照
	•	加上本轮 scratch provisional detector
	•	只在 scratch 副本上做 `register_boundary()`

⸻

17.5 Fusion adapter 的建议实现

Phase 1 不建议直接改 `OnlineBoundaryDetector` 核心逻辑，而建议包一层：

`CommittedDetectorState`
	•	只保存 committed 边界推进后的 detector / prior 状态

`ScratchDetectorState`
	•	每次 infer tick 从 committed 状态复制或重建
	•	只跑 active provisional window
	•	输出 provisional boundary events
	•	不污染 committed 状态

这样能最大程度复用现有代码，又不破坏 commit/revise 语义。

18. 当前代码接入时必须补的三个兼容层

18.1 Label space 兼容层
